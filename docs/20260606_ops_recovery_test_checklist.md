# 異常系リカバリ＋操作ログ 動作確認チェックリスト

> 対象機能: 確認メール再送 / 順位リンク再発行 ＋ 操作ログ（監査）。要件は `20260605_launchia_ops_recovery_requirements.md`。
> 実装・本番デプロイ済み（2026-06-06、commit `4dc7fbd`）。本書は実ログインでの手動 E2E 確認用。
> 作成: 2026-06-06

## 0. 準備（テストデータを作る）
- メールは **+エイリアス**で複数エントリを作ると便利（例 `setorosetorodev+t1@gmail.com`, `+t2`…）。
- 自分のプロジェクトの `/p/<slug>` か埋め込みウィジェットでテスト用メールを登録 → 「**確認待ち（未確認）**」エントリができる。
- その確認メール内 `/r/<token>` を一度開く → 「**確認済み**」エントリになる。
- ⚠️ 同一エントリへの再送/再発行は **5分クールダウン**。連続で試すときは別エントリを使うか5分待つ。

## A. 正常系：再送・再発行（`プロジェクト → 登録者一覧`）
- [ ] A-1 **未確認**エントリ行に「**確認メール再送**」が出る（確認済みは「順位リンク再発行」）
- [ ] A-2 「確認メール再送」→ 確認ダイアログOK → 「確認メールを再送しました」（緑）＋ 受信
- [ ] A-3 A-2 の**新リンク**を開く → 順位ページ表示（有効）
- [ ] A-4 再送**前の古いリンク**を開く → **`invalid_token`（失効）**。旧トークンが無効化される＝重要な仕様
- [ ] A-5 **確認済み**エントリで「**順位リンク再発行**」→ 新リンク有効・旧リンク失効

## B. 異常系・ガード
- [ ] B-1 同じエントリで続けてもう一度 → 「**少し時間をおいて再試行してください**」（429 クールダウン）。5分後は可能
- [ ] B-2 （UIでは状態で自動出し分けされるため通常起きないが）確認済みへ確認メール再送相当 → `already_confirmed`(409)
- [ ] B-3 エントリを**削除**後に一覧確認 → 削除済みは一覧に出ず再処理ボタンも出ない（宛先匿名化のため正しい）

## C. 操作ログ（`招待管理 → 操作ログ 🧾`、システム管理者のみ）
- [ ] C-1 A/B の操作が**新しい順**に並ぶ（日時／操作者＋役割バッジ／アクション／対象）
- [ ] C-2 アクションが日本語表示。対象メールが **`s***v@g***l.c*m` 形式でマスク**表示
- [ ] C-3 マスク横の「**表示**」で平文に**復号表示**
- [ ] C-4 C-3 後に再読込 → 「**宛先を復号表示**（`audit.reveal_email`）」の行が新たに記録（復号自体も監査）
- [ ] C-5 招待 承認/却下/発行・プロジェクト 作成/更新 もログに残る

## D. 権限（任意）
- [ ] D-1 **非管理者**で `/projects/audit-log` → `/projects` にリダイレクト
- [ ] D-2 （別開発者アカウントがあれば）他人のエントリ再処理 → **404（`project_not_found`）**（存在を漏らさない設計＝403 ではなく 404）。アカウント1つならスキップ

## E. サーバ/DB 側の裏取り（任意・Neon コンソール）
宛先は暗号化されており**コンソールでは読めない**（＝設計どおり。読むのは UI の「表示」から）:
```sql
SELECT created_at, actor_role, action, target_type, target_id,
       (target_email_enc IS NOT NULL) AS has_email
FROM launchia_admin_actions
ORDER BY created_at DESC
LIMIT 20;
```

## 最優先で見る3点（今回の肝）
1. **A-4**: 再送/再発行で**古いリンクが失効**する（トークン再発行が効いている）
2. **B-1**: **クールダウン**でメール爆撃を防げている
3. **C-2〜C-4**: 監査ログが**マスク表示・明示復号・復号も記録**（＝βポリシーの可逆暗号化が機能）
