# Launchia Phase 2 設計検討メモ

Phase 1 MVP の実装・動作確認（2026-05-28〜29）の過程で出た設計課題を、
Phase 2 以降の検討事項として記録する。Phase 1 では「暫定対処」または
「割り切り」で進め、本格対応は Phase 2 で行う。

---

## 1. エンドユーザー向けマイページ（複数登録・失念への対応）

### 課題
- 順位確認 URL（`/r/<token>`）が書かれたメールを**削除すると、順位を確認できなくなる**。
- ユーザーが**複数のウェイトリストに登録**している場合、個別の再送では非現実的（10 個なら 10 通）。
- ユーザーは登録した**アプリ名も「Launchia」自体も覚えていない**可能性が高い。「ちょっと前に登録したアレ」を辿る手段がない。

### 解（Phase 2/3）
- **メアドで名寄せした「マイページ」**。`launchia.net` でメアド入力 → ログインリンク/OTP → そのメアドで登録した**全プロジェクト + 順位**を一覧表示。
- アプリ名を覚えていなくても、**メアドさえあれば全登録を辿れる**。
- これは PRD（`20260527_launchia_Project.md`）の「マイ・ダッシュボード」に相当。

### 設計メモ
- DB: `launchia_waitlist_entries.email` で横断検索（**既存スキーマで対応可能**。後付けできる設計）。
- 認証: エンドユーザーにもログイン概念が必要 → 開発者と同じ仕組み（Magic Link または OTP）を流用。
- 認証方式の統一は要検討（開発者とエンドユーザーで同じ OTP 基盤を使うか）。

### Phase 1 での担保（割り切り）
- ウェイトリストの本質は「**リリース時に通知が届く**」こと。URL を紛失しても**メアドは DB に残る**ので、リリース通知は届く（致命的でない）。
- さらに **登録完了メール（後述・実装済み）** で「順位確認リンクを保管してください」と強調し、紛失をできるだけ防ぐ。

## 2. OTP コード認証（Magic Link の課題を解決）

### 課題（Phase 1 の Magic Link で顕在化）
- **(a) ブラウザ跨ぎ**: メールのログインリンクは OS の**デフォルトブラウザ**で開く。作業中のブラウザ（例 Chrome プロファイル A）と違うブラウザ（プロファイル B）で開くと、B にセッションができ、A では未ログインのまま。
- **(b) 複数プロファイル**: 開発者は Chrome プロファイルを複数使い分けるため、「どのブラウザでログインしたか分からなくなる」。
- **(c) プリフェッチ消費**: メールクライアント/セキュリティスキャナがリンクを先読みアクセスし、ワンタイムトークンを消費 → 本人のクリックが `invalid_token` になる（Phase 1 で遭遇）。

### 解（Phase 2 本命）
- ログインリンクの代わりに **6 桁の OTP コード**をメール送信 → **作業中のブラウザでコードを入力**。
- 効果:
  - ✅ ブラウザ跨ぎが消える（コードを入力したブラウザで完結）
  - ✅ プリフェッチ問題が根絶（リンクを踏まないのでスキャナの影響を受けない）
  - ✅ メールアプリ⇔作業ブラウザの行き来が不要
- 注意: **試行回数制限**（総当たり対策、例 5 回失敗でロック）+ レート制限。

### Phase 1 の暫定対処（実装済み）
- セッション Cookie を **30 日有効**にして再ログイン頻度を下げる。
- Magic Link を **15 分有効・複数回クリック可**に緩和（プリフェッチによる `invalid_token` を暫定回避。`auth.ts` の `consumeMagicLinkToken`）。

## 3. セッション管理画面（アクティブセッション一覧）

### 課題
- 複数のブラウザ/デバイスで、**どこにログイン中か把握できない**。

### 解（Phase 2）
- 設定画面に「現在ログイン中：Chrome / Windows（東京）… 」の一覧 + **個別ログアウト**（GitHub・Google 型）。

### 設計メモ
- 現状のセッションは **stateless な HMAC 署名 Cookie**（`lib/session.ts`）でサーバーに保存していない。
- セッション一覧を出すには、**セッションを DB 管理する設計変更**が必要（`launchia_sessions` テーブル等）。

---

## 4. 順位確認リンクの「登録解除」二段階化

### 課題
- 順位確認リンク（`/r/<token>`）には**順位閲覧と登録解除の両方**がある。
- このリンクを他人に渡すと、**相手が登録解除もできてしまう**。

### 解（Phase 2）
- 登録解除の前に**メアド入力などの確認ステップ**を挟み、リンク共有事故を防ぐ。

### Phase 1 の割り切り
- 「順位確認 URL を自分から他人に渡さない」運用。SNS でのスクショ共有（URL は写らない）は安全。

---

## 5. 濫用対策（レート制限・CAPTCHA）

### 課題
- メアドを入れれば本人にメールが飛ぶ仕組みのため、**大量送信でメール爆撃・サーバ/コスト負荷**のリスク。

### 解
- **レート制限**（要件書 5.4 記載）: magic-link 発行 1 メアド 5 通/時・1 IP 20 通/時、登録 1 IP 10 req/分 等。→ **M8 で Cloudflare Rate Limiting Rules** として実装。
- **CAPTCHA（Cloudflare Turnstile）**: bot による大量送信を根本封じ（Phase 2 候補）。

### Phase 1 の防御（一部実装済み）
- **招待制**が濫用の入口を絞っている（未登録メアド＋招待コードなし → メールを送らない）。
- レート制限はローカル未実装、M8（本番）で実装。

## Phase 1 / Phase 2 境界まとめ

| テーマ | Phase 1（実装済み / 割り切り） | Phase 2 以降（本格対応） |
| :--- | :--- | :--- |
| エンドユーザーの順位再確認 | 登録完了メールで「リンク保管」を強調。リリース通知はメアド保持で届く | **マイページ**（メアド名寄せで全登録一覧） |
| 開発者ログイン | Magic Link（15分・複数回可）+ 30日セッション | **OTP コード**（ブラウザ跨ぎ/プリフェッチ解決） |
| ログイン状態の把握 | （なし） | **セッション管理画面** |
| 順位確認リンクの解除 | リンクをシェアしない運用で割り切り | **解除の二段階確認** |
| 濫用対策 | 招待制で入口を制限 | **レート制限（M8）＋ CAPTCHA** |
