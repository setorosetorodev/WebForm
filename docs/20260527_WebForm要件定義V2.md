# 問い合わせフォーム Webパーツ化・マルチ環境対応プロジェクト 要件定義書（v2.0）

## 1. プロジェクト概要
### 1.1 目的
自社の各種ランディングページ（LP）に自由に埋め込める、デザインの自由度が高い「問い合わせフォーム（Webパーツ）」を自社開発する。
将来的なAzureおよびMicrosoft Dataverse（M365）への移行を見据えつつ、開発・プロトタイプ環境はCloudflareおよびNeonを用いたモダンかつ高速な環境で構築する。

### 1.2 目指す姿
- 既存のノーコードDBツール（Lister等）の制約を超え、LP側のデザインに一切干渉されない完全カプセル化されたWebパーツ。
- 特定のクラウドベンダーのランタイムに依存せず、ポータビリティの高いAPIバックエンド。
- ローカル/テスト環境と本番環境で、コードを書き換えることなくDBの接続先を切り替えられる柔軟なデータ層。

## 2. システム構成（アーキテクチャ）
本システムは、フロントエンド（Webパーツ）とバックエンド（APIプロキシ）が疎結合な構成とする。

### 2.1 フロントエンド（Webパーツ）
- **技術スタック:** Vanilla JavaScript (または軽量フレームワーク) / Web Components (Shadow DOM)
- **ホスティング:** Cloudflare Pages (将来的にも継続利用可)
- **特徴:** 
  - `<company-contact-form api-url="..." solution-id="..." require-consent="true">` のように、カスタムタグの属性で送信先API、ソリューションID、機能のON/OFF（同意チェックボックスなど）を動的に注入する。
  - LPの環境（HTML/CSS環境）に一切依存せず動作する。

### 2.2 バックエンド（APIプロキシ）
- **技術スタック:** Hono (マルチランタイム対応フレームワーク) / TypeScript
- **ホスティング (プロトタイプ):** Cloudflare Workers
- **ホスティング (本番環境):** Azure Functions
- **特徴:**
  - `Repositoryパターン`を採用し、ビジネスロジックとデータベースアクセスを分離する。
  - 環境変数により、接続先データベースおよび通知連携先を動的に切り替える。

### 2.3 データベース層
- **プロトタイプ環境:** Neon (Serverless Postgres) - 軽量ORM（Drizzle等）を用いてSQLでデータアクセス。
- **本番環境:** Microsoft Dataverse - Entra ID認証を経由し、OData(Web API)を用いてJSON形式でデータをPOST。

## 3. 機能要件
### 3.1 画面機能 (フロントエンド)
- **入力フォーム表示:** ユーザーが問い合わせ情報を入力するためのUIを表示する。デザインはモダンでトレンドを意識しつつ、拡張しやすい実装とする。
- **入力チェック (バリデーション):** 必須項目の抜け漏れや、メールアドレスの形式、電話番号（ハイフンあり）の形式等を送信前に検証する。
- **データ送信:** 送信ボタン押下時、入力データおよび「ソリューションID」をJSON形式でAPIへPOSTする。
- **ステータス表示:** 送信処理中のローディング表示を行う。通信エラー時は適切なエラーメッセージを表示する。
- **完了画面のインライン表示:** 送信完了後、外部ページへの遷移（リダイレクト）は行わず、Webパーツ内で「受け付けました」等の完了メッセージを表示して完結させる。
- **拡張性・カスタマイズ性:** 
  - 後から入力項目を追加・変更しやすいよう、フォーム項目を柔軟に管理できる仕組みを取り入れる。
  - 個人情報保護方針の提示機能を持つ。省スペース化のためリンク表示とし、チェックボックスを設けるかはHTMLの属性で切り替え可能な設計とする。

### 3.2 API機能 (バックエンド)
- **データ受信と検証:** フロントからのPOSTリクエストを受信し、データ型の検証（API側バリデーション）を行う。
- **データ保存:** 環境変数に基づき、NeonまたはDataverseへデータを保存する。
- **社内通知機能:** 問い合わせ受信時、Teams（Webhook / Workflows等）および MS Entra認証を介した社内メール経由で通知を送信する（プロトタイプ段階からM365エコシステムを活用）。
- **自動返信メール機能（動的テンプレート）:**
  - 受信した「ソリューションID」に基づいて、適切な返信メールテンプレートを選択する。
  - テンプレート内のプレースホルダー（例: `{{companyName}}`）に、入力された会社名や氏名を差し込み（リプレイス処理）、ユーザー宛にメールを送信する。

## 4. データ定義
### 4.1 入力項目
【必須・主要項目】
- 会社名
- 部署名
- 氏名
- メールアドレス
- 電話番号（ハイフンあり形式）
- 備考・自由記述欄（問い合わせ内容）

【隠し項目（システム用）】
- ソリューションID（LPごとにHTMLタグの属性から自動取得）

※ 将来的な項目の追加や変更を見越し、スキーマ（データ構造）は柔軟に拡張できる設計とする。

## 5. 非機能要件
### 5.1 セキュリティ
- **CORS設定:** 
  - プロトタイプ段階：テストを容易にするため全許可（`*`）とする。
  - 本番運用時：設定ファイル・環境変数から許可する自社LPドメインのリストを読み込み、それ以外からの通信をブロックする。
- **認証情報の秘匿:** データベースやメール配信用APIキー、M365（Entra ID）のクレデンシャルはすべて環境変数として管理し、フロントエンドには一切露出させない。

### 5.2 運用・保守性
- **コードの共通化:** Honoの採用により、Cloudflare環境とAzure環境でコアロジックを一致させる。
- **スキーマの一致:** プロトタイプ（Neon）のテーブル構造を、本番（Dataverse）の内部名にあらかじめ寄せて設計する。
